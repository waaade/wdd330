Week 10 Notes

Client-side Validation

  • Why client-side? Because if you only have server-side then the user won't know about the errors they submitted until the server sends its response. You can validate before sending to save time.
  • Why validate? So the data is inserted correctly. So their passwords are secure, protecting their data. And so no attacks compromise our site.
  • HTML5 has useful built-in validation: required, min, max, type, pattern
  • JavaScript validation is required for supporting old browsers that don't support the newer HTML5 stuff
  • Constraint validation API: validationMessage, validity (this one is important), willValidate. These can be used on just about any form element
  • On the validity thing: this has many properties, such as tooLong, typeMismatch, valueMissing. You gotta set things like minimums, maximums, and it'll validate them
  • Obvioulsy you can also validate forms without an API

Using Fetch

  • The Fetch API is a replacement for XMLHttpRequest. It's definitely easier.
  • Fetch can do POSTs too.
  • There are several response properties but the one used the most is probably "ok"
